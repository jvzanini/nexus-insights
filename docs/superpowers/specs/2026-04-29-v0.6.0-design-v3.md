# Spec v0.6.0 — Refazer Nexus Insights fiel ao Roteador (v3 final)

> **Versão:** v3 (após review #2)
> **Data:** 2026-04-29
> **Status:** consolidada, pronta pra geração de plan v1→v2→v3.

## 1. Motivação

João rejeitou v0.5.0 com 8 críticas concretas:
1. Filtros aplicam automaticamente (sem botão Aplicar).
2. Limite artificial de 90 dias no custom range.
3. Sem loading state visível.
4. Dashboard só com números, sem gráficos.
5. /perfil e /usuarios não batem com prints do Roteador.
6. Falta toggle de relatórios em /configuracoes (super_admin).
7. Dados inconsistentes em alguns relatórios.
8. Relatórios inúteis (especialmente leads-recebidos).

Mandou 7 prints do Roteador como referência visual obrigatória.

## 2. Princípios obrigatórios

| # | Princípio |
|---|-----------|
| P1 | Replicar Roteador fielmente — Source of Truth visual e arquitetural |
| P2 | Filtros multi-campo NUNCA aplicam automaticamente; pills de período (1-clique) podem |
| P3 | Sem limites artificiais (custom range pode cobrir todo o dataset) |
| P4 | Loading state em tudo — skeletons, não spinners soltos |
| P5 | Dashboard com gráficos visuais (Recharts) |
| P6 | Super_admin controla quais relatórios aparecem |
| P7 | Dados consistentes — validar com SQL direto |
| P8 | Sem código obsoleto |

### 3.3 Filtros — comportamento

#### Tipo: pills de período (1-clique aplica direto)
- Hoje | 7 dias | 30 dias | Personalizado
- Selecionar uma pill atualiza URL imediatamente, query roda.
- Custom abre date range picker SEM cap.

#### Tipo: filtros avançados multi-campo (botão Aplicar)
- Estado interno `draft` (em memória, mutado por inputs).
- Estado URL `applied` (na URL params).
- Botão "Aplicar filtros" envia draft → URL.
- Botão "Limpar" reset draft + URL.
- Indicador "X filtros pendentes" quando draft != applied.

#### Server-side helper:

```ts
export async function getEnabledReportKeys(): Promise<Set<string>> {
  const result = await pgPool.query(
    `SELECT value FROM app_settings WHERE key = 'platform.enabled_reports' LIMIT 1`,
  );
  if (!result.rows[0]) return new Set(REPORTS_CATALOG.map((r) => r.key));
  const value = result.rows[0].value;
  const arr = Array.isArray(value) ? value : (typeof value === 'object' && value && Array.isArray((value as any).value)) ? (value as any).value : [];
  return new Set(arr);
}
```

#### Sidebar respeita:
Server Component (já é) consulta `getEnabledReportKeys()` e filtra REPORTS_CATALOG antes de renderizar.

### 3.9 Errors & retry

- Componente `<ErrorState message retry>` com botão.
- Em pages: try/catch nos awaits Server Component, mostra ErrorState ou banner não-bloqueante.

### 3.10 Performance

- `Promise.all` paralelo onde aplicável (já é em conversas/page.tsx, dashboard).
- TTL 30s no cache reports.
- Polling 60s no dashboard (sem skeleton, silencioso).

### 3.11 Owner immutability

Server actions:
- `setUserActive(userId, active)`: bloqueia se `target.isOwner === true`.
- `deleteUser(userId)`: bloqueia se `target.isOwner === true`.
- `updateUser(userId, { platformRole })`: bloqueia se `target.isOwner === true && newRole !== 'super_admin'`.
- Retorna erro 403 com mensagem clara.

UI respeita (badge fixo, sem dropdowns/lixeira).
