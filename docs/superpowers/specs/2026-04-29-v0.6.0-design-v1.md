# Spec v0.6.0 — Refazer Nexus Insights fiel ao Roteador (v1)

> **Versão:** v1 (rascunho)
> **Data:** 2026-04-29
> **Origem:** rejeição do João sobre v0.5.0 ("uma porcaria, não fez nada do que pedi"). Reconstrução do zero alinhada ao padrão visual e funcional do Roteador Webhook Meta.

## 1. Motivação

João rejeitou v0.5.0 com 8 críticas concretas:

1. **Filtros aplicam automaticamente** sem botão "Aplicar" → frustrante quando quer ajustar múltiplos filtros antes de buscar.
2. **Limite artificial de 90 dias** no range custom → quer poder filtrar desde a primeira linha até hoje.
3. **Sem loading state visível** → clica e fica preto sem feedback até o conteúdo aparecer.
4. **Dashboard é só números** → quer gráficos visuais (linhas, barras), não só KPIs numéricos.
5. **/perfil e /usuarios não batem com Roteador** → mesmo depois de eu dizer que copiei, layout/spacing/dropdowns ficaram diferentes dos prints.
6. **Configurações falta toggle de relatórios** — super_admin precisa decidir quais relatórios estão disponíveis.
7. **Dados inconsistentes** — relatórios não refletem realidade (provavelmente bug de TZ/query).
8. **Relatórios são inúteis** — leads-recebidos não dá pra entender, nada útil pra rotina operacional.

Mandou 7 prints do Roteador como referência visual obrigatória.

## 2. Princípios obrigatórios

P1. **Replicar Roteador fielmente.** Roteador é o Source of Truth visual e arquitetural. Cada componente novo é uma adaptação de um componente equivalente do Roteador, não invenção.
P2. **Filtros NUNCA aplicam automaticamente** quando há múltiplos campos. Pills de período (Hoje / 7d / 30d) podem aplicar direto porque é 1 clique. Multi-campo (inbox + team + atendente + status + prioridade + período) tem botão "Aplicar".
P3. **Sem limites artificiais.** Custom range pode ir desde a primeira linha do banco até hoje. Sem cap de 90 dias.
P4. **Loading state em tudo.** Skeleton screens (não spinner solto) em todas as pages. Spinner em botões quando submetendo.
P5. **Dashboard tem gráficos.** Line chart, bar chart, donut — Recharts.
P6. **Super_admin controla quais relatórios aparecem.** Toggle ON/OFF em /configuracoes. Sidebar respeita.
P7. **Dados consistentes.** Verificar TZ em todas as queries. Validar cruzamentos.
P8. **Sem código obsoleto.** Apagar componentes/files não usados.

## 3. Escopo

### 3.1 Frente "Replicar Roteador" (visual + componentes)

- **Sidebar:** ajustes pra bater com Roteador (espaçamento, dividers, headers de seção)
- **/perfil:** 4 cards no padrão exato dos prints (Informações Pessoais com avatar+nome+Salvar, E-mail com Atual/Novo/Senha+Alterar, Senha com Atual/Nova/Confirmar+Alterar, Aparência com 3 toggles grandes)
- **/usuarios:** tabela com BadgeSelect inline pra Nível (Super Admin/Admin/Gerente/Visualizador com cores) e Status (Ativo/Inativo). Modal simples Criar/Editar (Nome, Email, Senha, Confirmar Senha, Nível, Toggle Ativo) — NÃO wizard. Botão lápis + lixeira nas Ações. Owner: badge fixo, sem dropdowns/lixeira.
- **/dashboard:** greeting + filtros (Empresa dropdown + Pills Hoje/7d/30d + Refresh) + 4 Stats Cards com comparison + Line Chart (Conversas por hora ou dia) + Top 5 cards (atendentes mais rápidos / mais aberto / inboxes) + Recent Deliveries-style table de "Últimas conversas".
- **Filtros do Conversas:** drawer/seção expansível com todos os filtros + botão "Aplicar filtros" + "Limpar".
- **Loading skeletons** em todas as pages que esperam dados.

### 3.2 Frente "Filtros decentes"

- **PeriodPills**: Hoje | 7 dias | 30 dias | Personalizado. Pills aplicam direto. Custom abre date range picker SEM limite (mín = 1 dia, max = todo o dataset).
- **AdvancedFilters** (componente novo): drawer ou painel inline que recebe todos os filtros multi-campo + botão "Aplicar". Estado interno (não-aplicado) vs URL state (aplicado). User edita estado interno, clica Aplicar, URL atualiza, query roda.
- **Botão "Limpar filtros"** em todo lugar que tem filtro avançado.

### 3.3 Frente "Loading states"

- Skeleton screens (matching the layout) em:
  - /dashboard (4 stats cards skeleton + chart skeleton + tables skeleton)
  - /usuarios (table rows skeleton)
  - /perfil (4 cards skeleton)
  - /relatorios/* (filtros skeleton + dados skeleton)
- Spinner em botões durante submit (Loader2 com animate-spin, label "Salvando…" / "Aplicando…")

### 3.4 Frente "Dashboard com gráficos"

Layout (top → bottom):
1. Greeting "Olá, {nome}" + data atual + Bell de notificações
2. Filtros (Empresa + Pills + Refresh)
3. 4 Stats Cards: "Conversas Recebidas", "Resolvidas", "Em Aberto", "Taxa de Resolução" — com comparison vs período anterior
4. Line Chart "Conversas por hora" (today) ou "Conversas por dia" (7d/30d) com 3 séries: Total / Resolvidas / Em Aberto
5. Top 3 cards lado a lado: Atendentes mais rápidos / Inboxes com mais aberto / Departamentos
6. Recent Conversations table (10 últimas) — Quando | Contato | Inbox | Atendente | Status

Gráficos usam Recharts (já instalado).

### 3.5 Frente "Configurações ativa/desativa relatórios"

- Em /configuracoes, nova seção "Relatórios disponíveis" (super_admin only).
- Lista de relatórios atualmente registrados (todos os 11 + Conversas + Mensagens não respondidas).
- Cada um com Switch ON/OFF + descrição.
- Persistido em `app_settings` chave `platform.enabled_reports` (Json array de keys).
- Sidebar consulta a chave e esconde itens não habilitados.
- Default: TODOS habilitados.

### 3.6 Frente "Dados consistentes"

- Verificar queries em `src/lib/chatwoot/queries/*` aplicando TZ corretamente em comparações de período.
- Cruzar conversas com mensagens corretamente (não duplicar quando há JOIN).
- Validar contagens (resolvidas, abertas) com SQL direto no Chatwoot pra ter ground truth.

### 3.7 Frente "Limpar código obsoleto"

- Apagar componentes não usados (busca por imports zero).
- Apagar pages que vão ser desativadas via toggle.
- Apagar mocks/dummies.

## 4. Trade-offs

- **Não vamos** consolidar 11 → 4 dashboards neste release (continua deferred). Toggle de habilitação cobre o problema de "tem 11 relatórios inúteis na sidebar".
- **Não vamos** trocar Recharts por outro stack.
- **Vamos** quebrar URLs de filtros antigos (period=7d / 30d) — middleware já tem REDIRECT_MAP estrutura pronta, mas continua vazio (rotas não foram movidas).

## 5. Critérios de aceitação

C1. /perfil renderiza idêntico ao print (4 cards, layout 2x2 desktop, stack mobile).
C2. /usuarios renderiza idêntico ao print (tabela + BadgeSelect inline + modal simples).
C3. /dashboard tem line chart visível com dados reais.
C4. Conversas tem botão "Aplicar filtros" — selecionar filtros sem clicar não dispara query.
C5. Custom range não tem cap de 90 dias.
C6. Toda page mostra skeleton durante load.
C7. /configuracoes tem aba/seção "Relatórios disponíveis" com toggles.
C8. Toggle OFF de um relatório esconde da sidebar imediatamente após salvar.
C9. Os 114 testes Jest existentes continuam passando.
C10. Login funciona (já tá; manter).

## 6. Riscos

R1. **Refatorar /usuarios sem regredir o caso do owner.** Manter `isOwner` → badge fixo, sem dropdowns. Garantir testes.
R2. **Mudar URL state dos filtros** pode quebrar URLs salvas. Aceitável (URLs ainda não são compartilhadas externamente).
R3. **Bug de TZ esconder dados** — verificar com SQL direto no Chatwoot antes de assumir bug.
