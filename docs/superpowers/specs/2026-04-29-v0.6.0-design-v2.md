### 3.5 Configurações ativa/desativa relatórios

- `/configuracoes` ganha card "Relatórios disponíveis" (super_admin only).
- Lista de relatórios registrados no projeto (catálogo central):

| Key | Label |
|-----|-------|
| `conversas` | Conversas |
| `mensagens-nao-respondidas` | Mensagens não respondidas |
| `csat` | CSAT |
| `leads-recebidos` | Leads Recebidos |
| `matrix-ia` | Matrix IA |
| `por-departamento` | Por Departamento |
| `por-estado` | Por Estado |
| `ranking-atendentes` | Ranking de Atendentes |
| `sla` | SLA |
| `status-conversas` | Status das Conversas |
| `tempos-resposta` | Tempos de Resposta |
| `volumetria` | Volumetria |

- Cada um com Switch ON/OFF.
- Default: TODOS habilitados.
- Persistência: `app_settings` chave `platform.enabled_reports` (Json: array de keys habilitadas).
- Após salvar, `router.refresh()` força sidebar a re-renderizar.
- **Sidebar respeita** consultando setting via Server Component.

Edge cases:
- Pelo menos 1 relatório precisa estar habilitado (validação na server action).

### 3.8 Errors & retry
- Cada page tem error boundary (component-level via try/catch + componente `<ErrorState retry={...} />`).
- Botão "Tentar novamente" chama re-fetch.

### 3.9 Performance
- `Promise.all` em todas as queries paralelas das pages.
- Cache 30s para dashboards (já implementado em queries/*.ts).
- Polling 60s (não bloqueia UI).

### 3.10 Tipos TypeScript
- `FilterState` strict typed.
- Sem `any` nas server actions.

### 3.11 Owner edge case
- Server action de update/delete bloqueia: nunca permitir desativar/excluir o último super_admin OU o owner.
- UI do /usuarios respeita: badge fixo, sem dropdowns/lixeira.
