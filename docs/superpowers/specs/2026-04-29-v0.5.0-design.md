## 3. Decisões arquiteturais

### 3.1 Timezone & locale como AppSetting

**Modelo:** 2 chaves novas em `AppSetting` (categoria `platform`):
- `platform.timezone` — string IANA, default `"America/Sao_Paulo"`
- `platform.locale` — string BCP-47, default `"pt-BR"`

**Fluxo:**
- Lazy default: `getPlatformTz()` retorna o default se chave não existe (sem criar registro). Isso evita necessidade de migration/seed adicional.
- Cache em memória (`Map`) com TTL 60s.
- Apenas `super_admin` edita via `/configuracoes`.
- Server action `updatePlatformSettings()` atualiza DB + chama `invalidatePlatformSettings()` + `invalidateReportsCache()` (porque mudança de TZ invalida ranges em cache).

**Validação:**
- Timezone: tentar `new Intl.DateTimeFormat("en-US", { timeZone: value })`. Se throw, rejeita.
- Locale: regex `/^[a-z]{2,3}(-[A-Z]{2,4})?$/` + tentar `new Intl.NumberFormat(value)`.

### 3.3 Filtros simplificados

`PeriodKey` reduzido para 4 valores: `hoje | semana_atual | mes_atual | custom`.

**Definições no platform tz:**
- `hoje`: 00:00:00 do dia atual → 24:00:00 do dia atual.
- `semana_atual`: segunda-feira 00:00 → próxima segunda 00:00 (ISO week).
- `mes_atual`: dia 1 00:00 do mês atual → dia 1 00:00 do próximo mês.
- `custom`: usuário escolhe `{ start: Date, end: Date }`. Validações:
  - `end > start`
  - Range máximo: 90 dias
  - Datas interpretadas como datas locais no platform tz

**UI (`<ConversasFilters />` e `<ReportFilters />`):**
- Pills horizontais. Mobile: `overflow-x-auto` + `snap-x`. Desktop: alinhamento à esquerda.
- Pill ativa: `bg-primary text-primary-foreground`. Inativa: `bg-muted text-muted-foreground`.
- Pill "Personalizado" com ícone Calendar:
  - Desktop: `<Popover>` com 2 calendários inline lado a lado + botão "Aplicar"
  - Mobile (≤640px): `<Sheet side="bottom">` com calendário stack vertical + bottom action bar
- Ao escolher custom, pill mostra range formatado: "01/Abr – 30/Abr"
- Reset (clicar pill ativa novamente): volta para `hoje`.

### 3.5 Senha temporária simples (#50)

Novo módulo `src/lib/utils/generate-temp-password.ts`:

```ts
import { randomInt } from "node:crypto";

const ALPHA = "abcdefghjkmnpqrstuvwxyzABCDEFGHJKMNPQRSTUVWXYZ"; // sem i, l, o, I, L, O
const DIGITS = "23456789"; // sem 0, 1
const POOL = ALPHA + DIGITS;

export function generateTempPassword(length = 8): string {
  if (length < 4) throw new Error("length mínimo 4");
  const chars: string[] = [
    ALPHA[randomInt(ALPHA.length)],
    DIGITS[randomInt(DIGITS.length)],
  ];
  for (let i = 2; i < length; i++) chars.push(POOL[randomInt(POOL.length)]);
  // Fisher-Yates para descorrelacionar posição
  for (let i = chars.length - 1; i > 0; i--) {
    const j = randomInt(i + 1);
    [chars[i], chars[j]] = [chars[j], chars[i]];
  }
  return chars.join("");
}
```

- Substitui o generator atual em `src/lib/actions/users/regenerate-password.ts`.
- Testes: 1000 amostras; verificar que nenhuma contém confundíveis e todas têm ≥1 letra E ≥1 dígito.

### 3.9 Mensagens não respondidas — nova tela (#56)

Path: `/relatorios/mensagens-nao-respondidas`

**Query:** `src/lib/chatwoot/queries/mensagens-nao-respondidas.ts`

```sql
WITH last_msg AS (
  SELECT DISTINCT ON (m.conversation_id)
    m.conversation_id, m.created_at, m.message_type, m.content
  FROM messages m
  ORDER BY m.conversation_id, m.created_at DESC
)
SELECT c.id, c.display_id, lm.created_at as last_incoming_at,
       NOW() - lm.created_at as waiting,
       lm.content as snippet,
       ct.name as contact_name, ct.phone_number,
       ix.name as inbox_name, tm.name as team_name, u.name as assignee_name
FROM conversations c
JOIN last_msg lm ON lm.conversation_id = c.id
LEFT JOIN contacts ct ON ct.id = c.contact_id
LEFT JOIN inboxes ix ON ix.id = c.inbox_id
LEFT JOIN teams tm ON tm.id = c.team_id
LEFT JOIN users u ON u.id = c.assignee_id
WHERE c.status = 0  -- open
  AND lm.message_type = 0  -- incoming
  AND c.account_id = $1
ORDER BY lm.created_at ASC  -- mais antigos primeiro
LIMIT 200;
```

**KPIs no topo:**
- Total
- Tempo médio de espera (avg)
- Mais antigo (max)

**Tabela:** Nome | WhatsApp | Estado | Departamento | Atendente | Aguardando há | Snippet | Ações.

**Filtros:** inbox, team, atendente. Sem período (é estado atual).

**Performance:**
- Cache 30s (live).
- Sugestão de índice (documentar; sem migration): `CREATE INDEX idx_messages_conv_created_type ON messages(conversation_id, created_at DESC, message_type)`.
- Se p95 > 2s em produção, recomendar contato com admin do Chatwoot para criar índice.

## 4. Trade-offs (com justificativa)

| Decisão | Alternativa | Por que escolhi |
|---------|-------------|-----------------|
| Lazy default em AppSetting (sem seed) | Migration que cria registros default | Lazy elimina necessidade de migration; valor default está no código (mais simples) |
| Heurística `last_activity_at` para "resolvido em" | Adicionar coluna `resolved_at` no Chatwoot | Não temos write access; heurística é boa o suficiente para 90% dos casos |
| Redirects via middleware | Apagar rotas antigas | Migração suave; URLs salvas continuam funcionando |
| Labels via JSON na query principal | Query separada por conversa | Reduz round-trips; performance OK com índice em taggings |
| Sem migration neste release | Adicionar tabela settings_v2 | YAGNI; AppSetting já modelado para isso |
| Recharts mantido | Trocar por Tremor/Visx | YAGNI; Recharts tá funcional |
