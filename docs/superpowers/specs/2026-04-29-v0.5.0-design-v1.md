# Spec v0.5.0 — Ajustes pós-feedback (v1)

> **Versão:** v1 (rascunho inicial)
> **Data:** 2026-04-29
> **Origem:** feedback do usuário após v0.4.0 — "dashboards uma bosta", relatórios "não servem pra nada".

## 1. Motivação

O usuário (owner João Zanini) abriu o produto após o deploy de v0.4.0 e relatou que:

1. O **dashboard atual não responde às perguntas operacionais reais** ("quantas conversas estão abertas agora? quantas mensagens não foram respondidas? quem é meu atendente mais rápido?").
2. Os **11 relatórios estão fragmentados, redundantes e sem foco operacional**. Vão consolidar em **4 dashboards robustos**.
3. O **relatório de Conversas** mistura "Última mensagem" (que ninguém usa) e esconde campos importantes (Documento, WhatsApp, Prioridade, Labels) em uma única coluna.
4. A **/perfil** não está fiel ao layout do Roteador Webhook Meta (referência arquitetural).
5. A **/usuarios** tem o switch de status no lugar errado (deveria ser dropdown), falta lápis de editar e o owner pode acidentalmente clicar em "regenerar senha" (foi exatamente o que aconteceu agora — usuário ficou trancado).
6. A **senha temporária** vinha com 16 caracteres + símbolos especiais, difícil de digitar quando o admin precisava entregá-la verbalmente.
7. As **Configurações** estão sem timezone/locale — todos os relatórios assumem UTC silenciosamente, dando confusão em horários.
8. As **queries do Chatwoot** comparam datas em UTC; "hoje" no painel não bate com "hoje" no Brasil (-3h), causando discrepâncias de até 3h no início/fim do dia.
9. Os **filtros** têm 6 opções (Hoje, Ontem, 7d, 30d, Mês atual, Mês anterior) — confuso e lento de usar no celular.
10. O **account switcher** aparece sempre, mesmo quando o usuário só tem 1 conta (sem utilidade, ocupa espaço).

## 3. Decisões arquiteturais

### 3.1 Timezone & locale como configurações de plataforma

Adicionar duas chaves ao `AppSetting` (categoria `platform`):
- `platform.timezone` (string IANA, default `America/Sao_Paulo`)
- `platform.locale` (string BCP-47, default `pt-BR`)

Apenas super_admin edita. Toda renderização de data/hora e toda construção de query de período passa por um helper central que lê essas configs (com cache em memória de 60s).

### 3.2 Helpers centrais

Criar `src/lib/datetime.ts` (puro, sem `'use client'`) com:
- `getPlatformTz(): Promise<string>` — lê AppSetting com cache.
- `getPeriodInTz(key: PeriodKey, tz: string): { start: Date; end: Date }` — calcula `start`/`end` em UTC mas considerando o "dia" como começa/termina no `tz` informado.
- `formatDate(date, opts)` / `formatDateTime` / `formatRelativeTime` — usa `Intl.DateTimeFormat` com locale e tz da plataforma.

Refatorar `src/lib/reports/period.ts` para delegar a esse helper.

### 3.3 Filtros simplificados

Reduzir `PeriodKey` de 6 → 4 valores: `hoje | semana_atual | mes_atual | custom`.
- "Semana atual" = segunda 00h00 → próxima segunda 00h00 (ISO week, no TZ da plataforma).
- "Custom" abre date range picker (popover com 2 calendários inline — desktop; sheet bottom — mobile).

UI: pills horizontais com scroll-x no mobile (sem dropdown). Selecionado tem `bg-primary text-primary-foreground`. Custom mostra ícone calendar à esquerda + range formatado.

### 3.4 Account switcher condicional

`Sidebar` recebe `accounts` array. Se `accounts.length <= 1`, não renderiza o componente (ocupa 0px, não fica espaço vazio).

### 3.5 Senha temporária simples

Substituir generator atual (16 chars com símbolos) por:
- 8 caracteres
- Apenas alfanuméricos
- Excluir confundíveis: `0`, `O`, `o`, `1`, `l`, `I`
- Garantir ≥1 letra e ≥1 número
- Função: `src/lib/utils/generate-temp-password.ts`

### 3.6 /perfil — 4 cards do Roteador

Substituir layout atual por 4 cards verticais empilhados (mobile) / 2x2 grid (≥md):

1. **Informações Pessoais** — avatar (uploader 96px), input nome, botão "Salvar alterações"
2. **E-mail** — exibe e-mail atual (read-only), input "novo e-mail", input "senha atual" (confirmação), botão "Alterar e-mail"
3. **Alterar Senha** — input "senha atual", input "nova senha", input "confirmar nova senha", botão "Alterar senha"
4. **Aparência** — 3 toggles grandes side-by-side (Escuro / Claro / Sistema) com ícone + label; selecionado tem ring + bg accent

Server actions já existem (`updateProfile`, `requestEmailChange`, `changePassword`, `updateTheme`); só refatorar UI.

### 3.7 /usuarios — fix UX do owner

Mudanças no `UserRow` da tabela:
1. Remover `Switch` de status; adicionar **dropdown** com 2 opções: "Ativo" / "Inativo" (Select com cores: ativo=verde, inativo=cinza).
2. Adicionar coluna "Ações" com botão **lápis** (ícone Pencil) que abre dialog de edição (nome + role + accounts + teams).
3. **Owner** (`isOwner=true`):
   - Status dropdown desabilitado (sempre "Ativo")
   - Botão regen-senha **escondido** (não disabled — escondido)
   - Botão de exclusão escondido
4. Botão "Adicionar usuário" passa de modal genérico → dialog estruturado com 3 abas: Informações → Acesso → Confirmação.

### 3.8 Dashboard novo (#55) — KPIs operacionais reais

Substituir cards atuais por:

**Linha 1 — KPIs primários (4 cards clicáveis):**
1. **Em Aberto** (status=open) — cor neutra, link para `/relatorios/conversas?status=open`
2. **Pendentes** (status=pending) — cor amarela, link para `/relatorios/conversas?status=pending`
3. **Resolvidas hoje** (status=resolved AND resolved_at=hoje) — cor verde, link para `/relatorios/conversas?status=resolved&period=hoje`
4. **Mensagens não respondidas** (open conversas onde última msg é incoming) — cor vermelha, link para `/relatorios/mensagens-nao-respondidas`

**Linha 2 — Top rankings (3 cards):**
1. Top 5 atendentes mais rápidos (menor first-response avg)
2. Top 5 atendentes com mais conversas em aberto
3. Top 5 inboxes (estados) com mais conversas em aberto

**Linha 3 — Volume:**
- Heatmap mantido (já é bom)

Cada Top 5 card:
- Lista compacta com avatar/iniciais + nome + métrica
- Mostra "Ver completo" → link para dashboard analítico correspondente

### 3.9 Mensagens não respondidas (#56) — nova tela

`/relatorios/mensagens-nao-respondidas` — lista de conversas onde:
- `c.status = 0` (open)
- A última `messages` tem `message_type = 0` (incoming)
- Filtros: período (com TZ Brasil), inbox, team, atendente
- Ordenação: tempo desde a última mensagem incoming (mais antiga primeiro = mais urgente)
- Colunas: Nome | WhatsApp | Estado | Departamento | Atendente | "Aguardando há" (relativeTime) | Última msg (snippet) | Abrir no Chatwoot
- KPI no topo: total + tempo médio de espera + mais antigo

Query nova: `src/lib/chatwoot/queries/mensagens-nao-respondidas.ts`.

### 3.10 Conversas redesign (#57)

Tabela atualizada — **9 colunas** (no lugar das 8 atuais, sem "Última mensagem"):

| # | Nome | WhatsApp | Documento | Estado | Departamento | Atendente | Status | Prioridade | Labels | Ações |

- **Nome:** truncado 200px com tooltip; vazio = "—"
- **WhatsApp:** `formatPhone()`; sem número = "—"
- **Documento:** CPF/CNPJ formatado; sem doc = "—"
- **Estado/Departamento/Atendente:** já existem
- **Status:** badge atual
- **Prioridade:** novo badge (urgent=vermelho, high=laranja, medium=amarelo, low=cinza, null=—)
- **Labels:** chips inline (até 2 visíveis + "+N" se mais); cores das labels do Chatwoot
- **Ações:** botão "Abrir no Chatwoot" (já existe)

Mobile: tabela vira cards (já existe `hidden md:table-cell` em algumas colunas — expandir).

Ajustes de query:
- Adicionar JOIN com `taggings + tags` para trazer labels
- Já tem priority via `c.priority` (não precisa novo)

### 3.11 Consolidação 11 → 4 dashboards (#58)

Reorganizar `/relatorios` em **4 dashboards** (cada um agrega 2-3 telas atuais):

| Novo dashboard | Substitui |
|----------------|-----------|
| **Operação** (`/relatorios/operacao`) | status-conversas + sla + tempos-resposta |
| **Atendentes** (`/relatorios/atendentes`) | ranking-atendentes + por-departamento |
| **Distribuição** (`/relatorios/distribuicao`) | por-estado + volumetria |
| **Origem & Resultado** (`/relatorios/origem-resultado`) | leads-recebidos + matrix-ia + csat |

Adicionalmente:
- **Conversas** continua como tela própria (`/relatorios/conversas`) — é uma listagem operacional, não um dashboard.
- **Mensagens não respondidas** continua tela própria (`/relatorios/mensagens-nao-respondidas`).

Cada dashboard novo tem:
- Header com filtros (período + inbox + team + atendente)
- Grid de cards/charts agregando o conteúdo das telas substituídas
- Tabs internas se for muito conteúdo (não mais que 3 tabs)

Sidebar reorganizada:
- Dashboard (geral)
- Operação
- Atendentes
- Distribuição
- Origem & Resultado
- — separador —
- Conversas
- Mensagens não respondidas

## 4. Trade-offs

- **Não vamos** apagar as rotas antigas imediatamente — vamos redirecioná-las pros novos dashboards (302 com query string preservada). Migração suave.
- **Não vamos** mudar o stack de charts (continua Recharts).
- **Vamos** quebrar URLs salvas dos relatórios antigos — aceitável dado o produto está em v0.x e ninguém de fora usa.
- **Vamos** remover features hoje no produto que não fazem sentido operacional (cards "comparativo período anterior" do dashboard) — simplifica.

## 5. Critérios de aceitação

1. Login + sessão continuam funcionando após mudança.
2. Owner não consegue clicar em "regenerar senha" sobre si mesmo.
3. Senha temporária gerada sempre tem 8 chars alfanuméricos sem confundíveis.
4. Em `/relatorios/conversas`, todas as 9 colunas aparecem corretamente preenchidas (com "—" quando não há dado).
5. KPI "Mensagens não respondidas" no dashboard, ao ser clicado, abre `/relatorios/mensagens-nao-respondidas` com lista preenchida.
6. Mudar timezone em `/configuracoes` para `UTC` faz "Hoje" no filtro mostrar dados de 0h UTC → 24h UTC (verificável).
7. Account switcher não aparece quando user só tem 1 conta.
8. Filtro de período no mobile (≤640px) não causa scroll horizontal nem corta texto.
9. Todos os 84 testes Jest existentes continuam passando.
10. Cobertura permanece ≥98% nos arquivos sob threshold.

## 6. Riscos & mitigação

- **Mudança de timezone retroativa pode quebrar filtros de cache em Redis** → invalidar cache reports na primeira ativação da feature.
- **Refatorar 11 telas em paralelo é arriscado** → entregar em 2 fases: foundation + UX shell primeiro (1-2 dias), dashboards novos depois (3-5 dias).
- **Query nova de "mensagens não respondidas"** pode ficar pesada → criar índice em `messages(conversation_id, created_at DESC, message_type)` se ainda não existir.

## 7. Plano de release

- v0.5.0-rc1: Foundation (TZ/locale/temp-password) + UX shell (perfil, usuarios, filtros, account switcher) → deploy
- v0.5.0-rc2: Dashboard novo + Mensagens não respondidas + Conversas redesign → deploy
- v0.5.0: Consolidação 11 → 4 dashboards → deploy
