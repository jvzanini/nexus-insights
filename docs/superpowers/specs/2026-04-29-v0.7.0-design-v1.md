# Nexus Insights — v0.7.0 — Design Spec (v1)

> **Status:** v1 — proposta inicial sujeita a review crítica.
> **Data:** 2026-04-29
> **Autor:** João Vitor Zanini (Claude — autônomo)
> **Topic:** ajustes UX após release v0.6.0 (rebranding + sidebar + filtros + tour + Matrix IA + tabela responsiva + pente fino Conversas)

---

## 1. Contexto e motivação

Após o deploy da v0.6.0, o usuário levantou diversos problemas de UX e qualidade no relatório principal (Conversas) e na navegação geral. Esta release foca em **polimento e qualidade de experiência**, sem adicionar novos relatórios. Os pontos foram dados de forma específica e detalhada — não há ambiguidade de escopo, apenas decisões de design.

### Princípios desta release

1. **Aproveitar telas grandes** — usuário tem monitor 27"; relatórios devem usar a tela toda, dentro de limites razoáveis.
2. **Filtros mais inteligentes, menos grosseiros** — manter (e ampliar) o poder, mas reduzir o ruído visual.
3. **Tour realmente útil** — cobrir todas as funcionalidades que o usuário pode descobrir, com layout não quebrado.
4. **Sincronização real** entre o toggle de Matrix IA em `/configuracoes` e todas as visualizações da plataforma.
5. **Sidebar limpa** — abandonar o tratamento "borda esquerda violet" para active state.
6. **Branding mais curto** — "Relatórios Inteligentes" no lugar de "Relatórios e insights dos atendimentos".

## 3. Decisões de design

### 3.1 [R1] Rebranding

- **Login** (`src/components/login/login-content.tsx` linha 75): substituir `"Relatórios e insights dos atendimentos"` por `"Relatórios Inteligentes"`.
- **Sidebar** (`src/components/layout/sidebar.tsx` linha 189): substituir `"Relatórios e insights"` por `"Relatórios Inteligentes"`.
- O título "Nexus Insights" permanece (ele é o nome da plataforma).
- O painel de branding lateral do login (`login-branding.tsx`) tem outro texto ("Relatórios e insights dos atendimentos.") — mantém-se, pois é claim hero, não tagline.

### 3.2 [R2] Sidebar — active state

**Hoje:** botão ativo recebe `bg-muted/50 text-violet-500 border-l-2 border-violet-500 pl-[10px]`. Visualmente: faixa violeta arredondada na borda esquerda. Aplicado idêntico para menu raiz e submenu — sem diferenciação. Usuário rejeitou.

**Decisão (A1 — pílula sólida sutil):**

- **Menu raiz ativo**:
  - `bg-violet-500/10` no botão inteiro (pílula), `text-violet-300` (dark) / `text-violet-700` (light), ícone violet.
  - Sem borda lateral, sem `pl-[10px]`.
  - Hover (não-ativo): `bg-muted/40` + `text-foreground`.
- **Submenu ativo**:
  - Diferenciar do raiz para não ficar duplicado.
  - **Indicador**: dot violeta 4 px (`•`) à esquerda do label, texto em violet, **sem fundo** (ou fundo `bg-violet-500/5` ultra sutil).
  - Hover: `bg-muted/30`, texto `text-foreground`.
- **Linha vertical do grupo expansível**: hoje `border-l border-border` no container dos children. Manter, mas reduzir opacidade.

**Por que A1**: padrão Linear/Vercel, limpo, sem ruído visual, e a pílula completa é mais clicável (Fitt's). Submenu com dot apenas evita "duas pílulas empilhadas" feio.

### 3.3 [R3] Largura útil

**Hoje:** `(protected)/layout.tsx` envolve `children` em `<div class="mx-auto max-w-7xl ...">`. `max-w-7xl` = 1280 px. Em 27"/2560 px, sobra 1280 px de espaço vazio dos dois lados.

**Decisão (C1):**

- Layout protegido **deixa de aplicar `max-w-7xl` no wrapper**. Em vez disso, cada página decide sua largura via classe específica.
- Criar utilitário/classe semântica:
  - `.page-shell-wide` → `max-w-[1600px]` (≈ 100rem) — usado em todos os relatórios (`/dashboard`, `/relatorios/*`, `/configuracoes/consumo`).
  - `.page-shell-narrow` → `max-w-7xl` (1280 px) — usado em páginas de admin denso e formulários (`/usuarios`, `/configuracoes`, `/configuracoes/llm`, `/perfil/*`).
- Padding lateral: `px-4 sm:px-6 lg:px-8 xl:px-10`.
- Padding top: `pt-16 sm:pt-8` (já existe; mantém-se para não colidir com botão hambúrguer mobile).
- Implementação: o wrapper do layout passa a ser `<main class="...">` sem `max-w-*`, e cada `page.tsx` (ou um `PageShell` componente) controla.

**Por que 1600 e não w-full**: 1600 px é suficiente para 15 colunas da tabela de Conversas se respirar bem. Acima disso, linhas de tabela ficam difíceis de ler horizontalmente (o olho perde linha). Páginas de admin com formulários densos não ganham nada com mais largura.

### 3.4 [R4] Pente fino — Conversas

#### 3.4.1 Coluna "Ações"

- Hoje: `defaultVisible: true`, mas `render: () => null` e o conteúdo da célula é tratado em `TableCell` com `if (col.key === "actions") <OpenInChatwoot />`. Funciona, mas é frágil.
- **Refator**: a coluna Ações define `render: (row) => <OpenInChatwoot accountId={..} displayId={row.display_id} />`. Remove o `if` no body. Mais limpo.
- **Botão**: aumentar área de toque mínima 32×32 px, ícone + label "Abrir" (já é assim). Manter o link `target="_blank"`.

#### 3.4.2 Labels

- Hoje (`labels-chips.tsx`): pinta cada label com cor determinística (hash do nome) + texto contrastante. Mostra `max=2` chips, resto vira `+N`.
- **Decisão**:
  - Sem cor — chip neutro `bg-muted/40 text-muted-foreground border border-border/40 rounded-full px-2 py-0.5 text-[11px]`.
  - Sem `+N` — mostrar **todas** as labels, com `flex-wrap`. Truncar cada label individual em `max-w-[140px]`.
  - Tooltip exibe nome completo se truncado.
  - Manter prop `max` (default `Infinity`) para o caso de querermos um cap em outro contexto futuro, mas não usar limite por padrão.

#### 3.4.3 Atributos customizados

- Hoje (`conversas-table.tsx` linhas 426–448): mostra `"N atributo(s)"` num pill, com tooltip contendo `key: value\n...`.
- **Decisão**:
  - Substituir por **lista inline**: `chave: valor · chave: valor` em chips/pílulas neutras (mesma estética das labels novas).
  - Cada par é um chip pequeno com `truncate max-w-[180px]`, tooltip com valor completo.
  - Wrap automático.
  - Quando 0 atributos: `—`.

#### 3.4.4 Ordenação

- **Hoje**: já existe multi-sort (click + shift+click) com indicador na header e botão "Limpar ordenação" no toolbar quando há sort ativo. Funcional, **mas não está descoberto pelo usuário**.
- **Decisão**:
  - Adicionar **chip "Ordenação · N"** no toolbar de filtros (linha sempre visível) — quando há sort ativo, mostra `N`.
  - Clicar no chip abre **Drawer de Ordenação** (ver §3.6) onde o usuário configura visualmente os critérios.
  - Header das colunas continua suportando click/shift-click (atalho avançado), com tooltip "Click ordena · Shift+click adiciona".
  - Documentar o atalho no tour (etapa 4).

#### 3.4.5 Densidade e responsividade

- Tabela já é horizontal-scrollable (`overflow-x-auto`). Com R3 (largura 1600 px), 15 colunas cabem sem scroll horizontal em monitor ≥ 1920 px.
- Manter `min-w-[XXX]` por coluna; ajustar `min-w-[120px]` para colunas com badge curto se necessário.

#### 3.4.6 "Conversas dessincronizado" / "parece sem conversa"

- Diagnóstico: em telas com filtros + tabela renderizada lado-a-lado, o usuário vê "0 conversas / Nenhuma conversa encontrada". Isso pode acontecer com filtros legítimos retornando vazio.
- **Investigação obrigatória durante implementação** (subagent dedicado): rodar a página com banco real (super_admin, sem filtros) e verificar se realmente há 0 ou se é bug de fetch/timezone/period-resolve. Se for bug, corrigir como parte de R4.
- Empty state já é decente. Adicionar dica útil quando `filterState !== EMPTY` no empty state: "Tente limpar filtros" (botão).

### 3.5 [R5] Filtros — busca interna + Selecionar todos + relabel

#### 3.5.1 MultiSelectCheckbox

- Adicionar **input de busca** no topo do popover (placeholder "Buscar..."). Filtra `options` por `option.name` case-insensitive.
- Adicionar **botão "Selecionar todos"** no header do popover, ao lado de "Limpar":
  - Quando `value.length === 0`: aparece "Selecionar todos".
  - Quando `value.length > 0 && < total`: aparece "Selecionar todos" (seleciona tudo).
  - Quando `value.length === total`: aparece "Limpar tudo".
- Quando usuário busca, "Selecionar todos" passa a aplicar **apenas aos resultados filtrados visíveis** (UX comum em multi-select com busca). Tooltip explica.
- Lista interna ganha `max-h-72` e `overflow-y-auto` com scrollbar customizada (já existe).

#### 3.5.2 Relabel "Equipe" → "Departamento"

- Em `advanced-filters.tsx`, o `MultiSelectCheckbox` de teams tem `label="Equipe"`. Mudar para `"Departamento"`.
- A coluna na tabela já é "Departamento". Consistência total.
- O modelo de dados continua `teams` (Chatwoot enum). Apenas a label visível muda.
- Tour copy também muda.

### 3.6 [R6] Filtros — toolbar + Drawer

#### 3.6.1 Toolbar (linha visível)

```
[ Hoje | Esta semana | Este mês | Todos | Personalizado ]   [ Buscar... ]   [ Filtros · 0 ]   [ Ordenação · 0 ]
```

- Layout: flex-wrap, gap 12 px.
- Busca ocupa flex-1 (ou min 320 px) na linha.
- Os dois chips ficam à direita (antes do "Aplicar filtros").
- Abaixo da linha principal, **chips de filtros aplicados** (rendered se `applied` tem alguma seleção):
  ```
  [ Departamento: 3 × ]  [ Atendente: João + 2 × ]  [ Status: Aberto × ]  [ Limpar tudo ]
  ```
- Cada chip remove daquele grupo ao clicar no X.

#### 3.6.2 Drawer "Filtros"

- Componente `Sheet` (lateral direita) ou `Dialog`. Usar **Sheet/Drawer right** para sensação de "atalho" e não interromper o fluxo.
- Largura 360 px (responsive: full em mobile).
- Conteúdo:
  - Cabeçalho com título "Filtros" e contador `N selecionados` + botão fechar.
  - **Seções colapsáveis** (default colapsadas, expandem ao clicar):
    - Caixa de entrada
    - Departamento
    - Atendente
    - Status
    - Prioridade
    - (Futuro: Labels, Atributos)
  - Cada seção mostra contador ao lado (`Atendente · 3`).
  - Dentro de cada seção: o `MultiSelectCheckbox` em modo "inline expandido" (não popover), com busca interna + Selecionar todos + lista.
  - Rodapé fixo: `[ Limpar todos ]  [ Aplicar (N pendentes) ]`.
- Ao clicar "Aplicar": fecha drawer + push URL + fetch.
- Pressionar ESC fecha sem aplicar.
- O drawer deve usar a mesma `FilterState` do componente atual (não duplicar).

#### 3.6.3 Drawer "Ordenação"

- Mesmo padrão Sheet, lateral direita.
- Conteúdo:
  - Header "Ordenação".
  - Lista vertical de critérios atuais. Cada item: `[≡] [Coluna ▾] [Asc/Desc toggle] [×]`.
    - `[≡]` — drag handle para reordenar prioridade (multi-sort).
    - `[Coluna ▾]` — select com colunas sortable disponíveis (não duplicar colunas já em uso).
    - `[Asc/Desc]` — toggle.
    - `[×]` — remove critério.
  - Botão `+ Adicionar critério` (abaixo da lista).
  - Rodapé: `[ Limpar ordenação ]  [ Aplicar ]`.
- O drawer manipula o mesmo estado `sortStack` do `ConversasTable` (passado via prop ou via `localStorage` compartilhado).

#### 3.6.4 Estado pendente vs aplicado

- Manter convenção atual: alterações ficam em `draft`, "Aplicar" promove para `applied` + URL push.
- A label "Todos os filtros aplicados / N pendente(s)" continua na linha do toolbar.

#### 3.7.3 Tour overlay — fix de layout

- Bug observado nos screenshots: popover passa por cima do conteúdo + botão "Próximo" fora do componente em algumas resoluções.
- Causa raiz: `POPOVER_EST_HEIGHT = 200` é estimativa fixa; quando o conteúdo (textos longos) + 2 ações + footer estoura 200 px, o popover é posicionado errado e "transborda".
- **Fix**:
  - Usar `ResizeObserver` no próprio popover para medir altura real e reposicionar quando muda.
  - Ou, mais simples: aumentar `POPOVER_EST_HEIGHT` para 280 px e clamp final que **só posiciona depois que sabe a altura real** (medir via ref).
  - Garantir que botões caibam: aumentar `width` para 360 px e empilhar Voltar/Próximo abaixo dos dots em viewports ≤ 480 px.

### 3.8 [R8] Sincronização Matrix IA

#### 3.8.1 Estado atual

- `appSetting.value` `reports.include_matrix_ia` controla se inbox 31 aparece nos relatórios.
- `shouldExcludeMatrixIA()` retorna `true` para não-super_admins quando flag é `false`.
- `buildBaseFilter` aplica `c.inbox_id <> 31` nas queries Chatwoot.
- **Brecha 1**: `getInboxes()` (meta-cache) retorna **todas** as inboxes incluindo 31, mesmo para usuários não-super_admin. Resultado: inbox 31 aparece nos dropdowns de filtro como "00-Matrix IA".
- **Brecha 2**: drill-down sheets, charts (status pie, dow, heatmap, ranking) — verificar se cada query passa `excludeMatrixIA` ou se herda do `buildBaseFilter` default.

#### 3.8.2 Decisão

- Criar helper `getInboxesForUser(accountId, user)` que envolve `getInboxes(accountId)` e:
  - Se `user.platformRole === 'super_admin'` → retorna todas.
  - Senão e `appSetting.reports.include_matrix_ia === false` → filtra `id !== 31`.
- Substituir todos os usos de `getInboxes(accountId)` em rotas/server-components por `getInboxesForUser(accountId, user)`.
- **Auditoria**: rodar grep `getInboxes(` e atualizar todos os call sites em páginas autenticadas.
- **Drill-downs / charts**: garantir que toda query de `dashboard-data.ts`, `dashboard-drill-down.ts`, `dashboard-kpis.ts`, `por-estado.ts`, `por-departamento.ts`, `ranking-atendentes.ts`, `tempos-resposta.ts`, `volumetria-*.ts`, `status-distribution.ts`, `leads-recebidos.ts`, `home-summary.ts`, `mensagens-nao-respondidas.ts` recebe e respeita `excludeMatrixIA`. Onde não respeita, corrigir.
- **Listas top-N**: se algum chart/lista mostra "Top inboxes" ou similar, filtrar 31 da lista quando aplicável.
- **Card Matrix IA específico** (`matrix-ia-content.tsx`): este card é **somente sobre** inbox 31. Quando flag é OFF e user não é super_admin: ocultar o card / a tab (não exibir).

## 5. Riscos e mitigações

| Risco | Mitigação |
|-------|-----------|
| Drawer mobile pode conflitar com bubble Nex | Drawer usa z-index 1900; bubble fica em z 2000. Verificar e ajustar. |
| `getInboxesForUser` quebra cache atual (compartilhado entre usuários) | Cache continua sendo de `getInboxes` (meta sem filtro); o filtro é aplicado em memória após retorno. Sem invalidação extra. |
| `max-w-[1600px]` "afina" demais em telas 1920×1080 (sobra pouco) | Em 1920 px: 1600 px de conteúdo + 160 px padding cada lado = OK. |
| Popover do tour ainda quebrar em algum caso edge | Testar em viewports 1440, 1920, 2560, e mobile 390. |
| Inbox 31 escondida pode confundir admin que esperava ver | Toast já avisa após toggle. Documentar em runbook. |

11. Toggle ON → manager passa a ver normalmente.
